## HW 问题收集

列举在 HW 1、HW1.1 过程里，你所遇到的 2~3 个通过自己学习已经解决的问题，和 2~3 个尚未解决的问题与挑战。

### 已解决

1. **啥是”标量 derived store“ 有啥作用？**
   1. **上下文**：Coding Agent 说：”UI 的棋盘、输入、Undo/Redo、胜利判断、分享编码全部直接读取领域对象公开接口；允许保留少量标量 derived store，但不再生成板级 view model。“ 
   2. **解决手段**：直接询问 CA + 查看 Svelte 官方文档。我理解了 `derived` store 就是基于一个或多个原有 store 派生出的只读状态。“标量”就是指它衍生出来的是简单的布尔值或数字（例如 `canUndo`、`isWon`、`conflictsCount`），而不是把整个庞大的棋盘对象又重新包装一遍。它的作用是让 UI 组件按需订阅最小粒度的数据，避免不必要的重新渲染，同时不破坏领域模型的核心地位。

2. **如何处理 Svelte 响应式机制与纯 JS 领域对象（OOP）的脱节问题？**
   1. **上下文**：在 HW1 中，我设计了完美的 `Game` 和 `Sudoku` 类，但在接入 Svelte 时，发现即使执行了 `game.guess()`，网页上的数字也不更新。
   2. **解决手段**：深入学习 Svelte 的脏检查机制后，发现 Svelte 只能监听到赋值操作（`=`）。直接 mutate 深层对象属性是无效的。因此我采用了 **Store Adapter (适配器) 模式**，在 `domainAdapter.js` 中封装领域实例，并在每次操作后手动调用 `set()` 方法抛出一个全新的 UI 状态快照，成功打通了底层逻辑与视图更新的桥梁。

3. **规则校验应该放在 UI 层还是 Domain 领域层？**
   1. **上下文**：在最初的版本中，我把九宫格的冲突检测算法和“禁止修改原生题目”的逻辑写在了 `Store Adapter` 里面，导致 Code Review 提出“Sudoku 没有真正承载数独规则”。
   2. **解决手段**：重新审视了领域驱动设计（DDD）原则。UI 和 Store 只是业务的“表现”，真正的业务约束（如数独不能有行列冲突）必须内聚在领域对象内。我重构了 `Sudoku.js`，将 `initialGrid` 记忆、越界检查和冲突检测全部收敛到了 `guess` 方法内部，并通过 `throw new Error()` 严格捍卫领域规则。

### 未解决 / 挑战

1. **这个 `sameArea` 有啥用啊？**
   1. **上下文**：`src/components/Board/index.svelte` 中有这样一段代码：
      ```javascript
      sameArea={$settings.highlightCells && !isSelected($cursor, x, y) && isSameArea($cursor, x, y)}
      ```
   2. **尝试解决手段**：查阅旧代码发现它主要用于视觉反馈。当玩家选中一个格子时，UI 需要高亮与该格子位于同一行、同一列、以及同一个 3x3 宫格内的所有其他格子（通常是浅色高亮），以辅助玩家肉眼排查冲突。这部分属于纯粹的 View 层逻辑，但目前尚未与新重构的 Domain 模型完全隔离解耦。

2. **如何处理领域层抛出的 Error 而不导致整个前端应用崩溃？**
   1. **上下文**：在修复 HW1.1 缺陷时，我听从建议在 `Sudoku.guess()` 中加入了严格的 `throw new Error("同行存在冲突")` 机制来替代无声的 `console.warn`。
   2. **尝试解决手段/挑战**：虽然领域对象变得安全了，但随之而来的挑战是当用户在界面上输入一个错误数字时，Svelte 前端接收到这个 Error 如果没有被 `try...catch` 妥善包裹，会导致整个组件树卡死。如何在 Store Adapter 中捕获这些领域异常，并将其转化为用户友好的 UI 提示（比如弹窗或红色震动），目前在架构链路的异常处理机制上还没有找到完美的统一实践方案。
