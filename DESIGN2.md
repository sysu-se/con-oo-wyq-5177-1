# 领域对象接入与响应式机制设计

## A. 领域对象如何被消费

**1. View 层直接消费的是什么？**
View 层（Svelte 组件）**没有**直接消费 `Game` 或 `Sudoku` 的类实例，而是消费了一个名为 `gameStore` 的 Store 适配器（Adapter）。这个适配器内部包裹了真实的领域对象，并对外暴露了响应式接口。

**2. View 层拿到的数据是什么？**
View 层通过订阅 `$gameStore`，拿到的是每次操作后经过计算的**纯 UI 状态快照**，包含：
* `grid`: 当前最新盘面的二维数组。
* `initialGrid`: 开局题目数据（用于区分原生数字和玩家填入的数字，防止篡改）。
* `conflicts`: 冲突检测矩阵（用于将违反数独规则的同行/同列/同宫格子标红）。
* `canUndo` / `canRedo`: 布尔值，用于控制撤销/重做按钮的禁用状态。

**3. 用户操作如何进入领域对象？**
* **光标移动：** 点击格子触发 `cursor.set()` 更新 Svelte 的全局光标状态。
* **填入数字：** 点击小键盘触发 `gameStore.guess(row, col, value)`。该方法内部会调用纯领域对象的 `gameInstance.guess()` 执行核心业务逻辑。
* **撤销/重做：** 点击 UI 按钮调用 `gameStore.undo()`，同样转发给底层的 `gameInstance` 处理。

**4. 领域对象变化后，Svelte 为什么会更新？**
当用户通过 `gameStore.guess()` 等入口操作后，底层 `Game` 实例的状态改变。此时，Store 内部会调用 Svelte 原生的 `set(getUIState())` 方法。因为传入了一个全新的状态对象字面量，Svelte 的响应式系统捕获到了数据引用的变化，从而触发订阅了 `$gameStore` 的 `Board` 和 `Cell` 组件重新渲染。

## B. 响应式机制说明

**1. 依赖的响应式机制：**
本方案核心依赖了 Svelte 3 的 `writable store` 机制，并配合组件内的 `$store` 语法糖实现自动订阅与解绑。

**2. 响应式暴露的状态 vs 留在内部的状态：**
* **响应式暴露：** 上述提到的 UI 渲染所需状态（`grid`, `conflicts` 等）都被打包成响应式对象暴露给 View 层。
* **留在内部（不可见）：** 真正的领域对象实例（`Game` 和 `Sudoku`），以及用来实现撤销/重做的极其敏感的 `history` 和 `redoHistory` 历史栈。这些被严格封装在 Adapter 的闭包内，View 层无法直接触碰。

**3. 如果直接 mutate 内部对象，会出现什么问题？**
如果不使用 Store Adapter 抛出新对象，而是直接在组件里执行 `game.getSudoku().grid[y][x] = value`（直接 Mutate 深层属性），因为没有发生**重新赋值运算（`=` 改变对象引用）**，Svelte 的 `$:` 语句和模板渲染无法感知到深层数据的变化，会导致极其严重的“底层数据已修改，但网页界面死活不刷新（脱节）”的 Bug。

## C. 对 HW 1 的改进说明

**1. 相比 HW1 改进了什么？**
* **构造与获取的防卫式克隆：** 修复了 HW1 中 `Game` 暴露内部状态的致命缺陷。现在的 `Game` 在构造接收和 `getSudoku()` 时，全部采用深拷贝（`.clone()`）切断外部引用，夺回了对象状态的所有权。
* **领域规则收敛：** `Sudoku.guess` 不再是单纯的数据容器，加入了针对坐标越界、以及原生题目的“防篡改”拦截逻辑。
* **加入 Adapter 隔离层：** 实现了 Model 与 View 的彻底解耦。

**2. 为什么 HW1 中的做法不足以支撑真实接入？**
HW1 的设计像是一个“带少量面向对象包装的裸数据结构”。如果直接将 HW1 的 `Game` 实例绑定到 UI 上，前端的 JS 脚本可以轻易绕过 `guess` 方法，直接修改 `game.currentSudoku`。这会瞬间导致“历史记录栈”与“当前盘面”彻底脱轨，Undo/Redo 系统将立刻崩溃。

**3. 新设计的 Trade-off (权衡) 评估：**
* **牺牲内存换取稳定性：** 为了保证不可变数据和历史栈的绝对安全，每次落子、每次读取都在进行二维数组的深度克隆。在超大型应用中可能有性能瓶颈，但在 9x9 的数独领域中，这种内存开销微乎其微，却换来了极高且极安全的架构稳定性，是以空间换架构的典范实践。
* **彻底的单向数据流：** 展示组件（`Cell`）脱离了对 Store 的直接依赖，变成纯粹的受控组件（Dumb Component），极大提升了代码的可复用性。
